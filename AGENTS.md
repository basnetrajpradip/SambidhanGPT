## Project Identity

**SambidhanGPT** is an AI-powered legal document Q&A system with PDF citation highlighting.
It serves Nepali legal professionals, students, and citizens who need to query dense legal PDFs in natural language.

**Core principle:** Every answer MUST be grounded in the uploaded document. Never generate legal content from model memory. If the context doesn't contain the answer, say so.

---

## Model

- **LLM:** `gemini-2.5-flash` (Google AI Studio) вЂ” used for answer generation, clause extraction, and question suggestion
- **Embeddings:** `gemini-embedding-001` (Google) вЂ” **3072 dimensions** (`output_dimensionality: 3072`)
- **DO NOT** switch models without updating this file and re-running embedding tests
- **DO NOT** change embedding dimensions without dropping and re-creating the `chunks` table вЂ” pgvector will reject mismatched dimensions silently
- **temperature:** `0` for Q&A, `0.2` for clause extraction and suggestions

## Critical Rules for Agents

### вљ пёЏ NEVER DO THESE

- **NEVER** generate a legal answer from model memory вЂ” only from retrieved document chunks
- **NEVER** return an answer without a `citations` array, even if it is empty
- **NEVER** store entire PDFs as a single embedding вЂ” always chunk first
- **NEVER** mix up `char_offset_start` / `char_offset_end` вЂ” citation highlighting depends on byte-accurate offsets
- **NEVER** skip the contextual compression step вЂ” it reduces token waste and improves precision
- **NEVER** change the embedding model or dimensions without dropping and re-creating the `chunks` table
- **NEVER** expose one user's documents to another user
- **NEVER** write raw `CREATE TABLE` SQL manually вЂ” schema lives in `db/schema/`, migrations are generated by `drizzle-kit generate`
- **NEVER** use `drizzle-kit push` in production вЂ” always use `drizzle-kit migrate`
- **NEVER** call raw SQL for similarity search вЂ” use Drizzle's `cosineDistance` helper

### вњ… ALWAYS DO THESE

- **ALWAYS** include `page_number`, `char_offset_start`, `char_offset_end` when storing chunks
- **ALWAYS** use `responseMimeType: 'application/json'` with Gemini for structured outputs
- **ALWAYS** set `output_dimensionality: 3072` when calling `gemini-embedding-001` вЂ” do not rely on the default 3072
- **ALWAYS** validate pgvector index type is `ivfflat` or `hnsw` вЂ” never do brute-force search in production
- **ALWAYS** handle PDF parsing failures gracefully вЂ” return a clear error, do not crash the ingestion pipeline
- **ALWAYS** keep conversation history injection bounded to `MAX_CONVERSATION_TURNS`
- **ALWAYS** test chunking changes with `test/chunking.test.ts` before touching other services

## Agents

These are specialized task agents in `backend/agents/`. All agent files are `.ts`.

### 1. рџ”§ IngestionAgent

**File:** `agents/ingestion-agent.ts`
**Trigger:** After PDF upload via `POST /api/documents/upload`
**Responsibility:**

1. Parse PDF в†’ extract raw text + page/position metadata using `pdfjs-dist`
2. Chunk text using LangChain `RecursiveCharacterTextSplitter` (500 tokens, 50 overlap)
3. Embed each chunk using `gemini-embedding-001` with `output_dimensionality: 3072`
4. Store in `chunks` table via Drizzle: `(document_id, content, page_number, char_offset_start, char_offset_end, embedding)`
5. Trigger `ClauseAgent` and `SuggestionAgent` after ingestion completes

**Output:** `{ document_id: string, chunk_count: number, status: 'complete' | 'failed' }`

**Error handling:** If any chunk fails to embed, log and skip it. Do not abort the entire ingestion. Report partial failures in the output.

---

### 2. рџ’¬ RAGAgent

**File:** `agents/rag-agent.ts`
**Trigger:** User submits a question via `POST /api/chat`
**Responsibility:**

1. Embed the user query using `gemini-embedding-001` with `output_dimensionality: 3072`
2. Run cosine similarity search via Drizzle's `cosineDistance` helper (`TOP_K=5` chunks)
3. Apply contextual compression: filter each chunk to most relevant sentences only
4. Assemble prompt: `[system prompt] + [compressed context] + [conversation history] + [user question]`
5. Call `gemini-2.5-flash` with `responseMimeType: 'application/json'`
6. Parse and return `{ answer, citations[] }`

**Gemini Prompt вЂ” Q&A System Prompt** (in `llm/prompts/qa-prompt.ts`):

```
You are SambidhanGPT, a legal document assistant specialized in Nepali constitutional and civil law.

Rules:
1. Answer ONLY from the provided document context. Never use outside knowledge.
2. If the answer is not in the context, respond: "This information is not found in the uploaded document."
3. Cite specific article numbers, section numbers, or clause names when present.
4. Be precise and concise. Avoid unnecessary elaboration.
5. Return your response as JSON:
{
  "answer": "...",
  "citations": [{ "chunk_id": "...", "page": N, "excerpt": "..." }]
}
```

**Output:** `{ answer: string, citations: Array<{ chunk_id: string, page: number, char_offset_start: number, char_offset_end: number, excerpt: string }> }`

---

### 3. рџ“‹ ClauseAgent

**File:** `agents/clause-agent.ts`
**Trigger:** After IngestionAgent completes
**Responsibility:**

1. Sample representative chunks spread evenly across the document (every N-th chunk)
2. Send to `gemini-2.5-flash` with clause extraction prompt (`temperature: 0.2`)
3. Store extracted clauses in `clauses` table via Drizzle with `type`, `title`, `excerpt`, `page_number`

**Clause types to detect:** `indemnity`, `termination`, `liability`, `payment_terms`, `jurisdiction`, `amendment`, `definitions`, `penalties`

**Gemini Prompt** (in `llm/prompts/clause-prompt.ts`):

```
You are a legal document analyzer. From the provided text, identify and extract key legal clauses.
Return JSON: { "clauses": [{ "type": "...", "title": "...", "excerpt": "...", "page": N }] }
Clause types: indemnity, termination, liability, payment_terms, jurisdiction, amendment, definitions, penalties.
Return only clauses explicitly present in the text. Omit any type not found.
```

**Output:** `{ document_id: string, clauses: Array<{ type: string, title: string, excerpt: string, page: number }> }`

---

### 4. рџ’Ў SuggestionAgent

**File:** `agents/suggestion-agent.ts`
**Trigger:** After IngestionAgent completes
**Responsibility:**

1. Select 5 chunks spread evenly across document (not clustered at the start)
2. Send to `gemini-2.5-flash` with suggestion prompt (`temperature: 0.2`)
3. Return 5вЂ“7 specific, answerable questions as a JSON array
4. Store each question as a row in `suggestions` table via Drizzle

**Gemini Prompt** (in `llm/prompts/suggestion-prompt.ts`):

```
You are a legal research assistant. Based on the following excerpts from a legal document,
generate 5 to 7 specific, answerable questions a user might ask about this document.
Rules:
- Questions must be specific to these excerpts, not generic legal questions
- Questions should be concise (under 15 words each)
- Questions should cover different aspects of the document
Return ONLY a JSON array: ["question 1", "question 2", ...]
```

**Output:** `{ document_id: string, suggestions: string[] }`

### 5. рџ”Ќ CitationAgent (Frontend)

**File:** `frontend/src/services/citation-agent.ts`
**Trigger:** After RAGAgent returns citations
**Responsibility:**

1. Receive `citations[]` from Q&A response
2. For each citation, use `page_number` + `char_offset_start` + `char_offset_end` to compute PDF highlight coordinates
3. Pass highlight regions to PDF viewer annotation layer
4. Scroll PDF viewer to the first citation's page on answer load

**Libraries:** `react-pdf` for rendering, `pdfjs-dist` for text layer coordinate mapping

## Drizzle ORM Rules

- **Schema is the source of truth.** All table definitions live in `db/schema/`. Never write raw DDL.
- **Generate migrations** with `npx drizzle-kit generate` after any schema change.
- **Apply migrations** with `npx drizzle-kit migrate` вЂ” never `push` in production.
- **Similarity search** uses Drizzle's built-in `cosineDistance` from `drizzle-orm`:

```ts
import { cosineDistance, desc } from 'drizzle-orm'
import { eq } from 'drizzle-orm'

const results = await db
  .select()
  .from(chunks)
  .where(eq(chunks.documentId, documentId))
  .orderBy(desc(cosineDistance(chunks.embedding, queryVector)))
  .limit(TOP_K)
```

- **Vector column** is defined as `vector("embedding", { dimensions: 3072 })` in `db/schema/chunks.ts`
- **ivfflat index** is created in the initial migration вЂ” do not remove it

## Chunking Strategy

```
Chunk size:    500 tokens
Overlap:       50 tokens
Splitter:      RecursiveCharacterTextSplitter
Split order:   ["\n\n", "\n", ". ", " ", ""]
```

Each chunk stored with:

- `content` вЂ” the raw text
- `page_number` вЂ” PDF page it came from
- `char_offset_start` вЂ” character position in the full extracted text
- `char_offset_end` вЂ” end character position
- `embedding` вЂ” 3072-dim vector (`gemini-embedding-001` with `output_dimensionality: 3072`)

**Why these settings matter:** The `char_offset` fields are the only way the frontend can highlight the exact text in the PDF viewer. If these are wrong, citations will highlight the wrong region. Test carefully with `test/citation.test.ts`.

## Prompt Engineering Notes

- Keep all prompts in `llm/prompts/*.ts` вЂ” never inline prompts in controllers or services
- Always use `responseMimeType: 'application/json'` with Gemini for structured outputs
- Always set `temperature: 0` for legal Q&A вЂ” determinism matters more than creativity
- For clause extraction and suggestions, `temperature: 0.2` is acceptable
- Max output tokens for Q&A: `1500`
- Max output tokens for suggestions: `300`
- Max output tokens for clause extraction: `800`
